# Building PromptAmp from source

Written for an AMO reviewer, who must be able to reproduce the submitted
artifact byte-for-byte from this repository, on a clean machine, without
guessing anything.

## Environment

| | Version |
| --- | --- |
| Node.js | 22.x (tested on 22.20.1) |
| pnpm | 9.x |
| OS | Any; the toolchain has no native dependencies |

pnpm is required rather than npm because `pnpm-lock.yaml` is the committed
lockfile. Installing with npm resolves a different dependency graph and the
output will not match.

## Build

```bash
pnpm install --frozen-lockfile
pnpm build:firefox
```

Output: `.output/firefox-mv3/`

To produce the uploadable zip instead:

```bash
pnpm zip:firefox
```

Output: `.output/promptamp-<version>-firefox.zip`

## Verifying the whole thing

```bash
pnpm typecheck   # tsc --noEmit, strict
pnpm lint        # ESLint, including the rule that blocks key access from content scripts
pnpm test        # 270 unit tests
pnpm e2e         # 79 end-to-end tests against a real loaded extension
```

`pnpm e2e` builds first and drives a local page in `playground/`. It makes no
network requests and needs no API key: a mock provider (`lib/providers/mock.ts`)
serves every response deterministically.

## What is in the source zip

All of it. There is no minified, obfuscated, transpiled, or generated code in
the repository — every file under `lib/`, `entrypoints/`, `e2e/`, and `tests/`
is the original TypeScript as written.

The only generated artifacts are:

| Path | Generated by | Committed? |
| --- | --- | --- |
| `.output/` | `wxt build` | No |
| `.wxt/` | `wxt prepare` | No |
| `public/icon/*.png` | `pnpm icons`, from `assets/icon.svg` | Yes — a store build must not depend on downloading a browser |
| `store/screenshots/` | `pnpm shots` | No |

## Where to look

A reviewer's time is best spent on these five, in this order:

### 1. No remote code — `entrypoints/background.ts`, `lib/insertion/main-world.ts`

There is no `eval`, no `new Function`, and no script loaded from a URL. The one
call to `chrome.scripting.executeScript` injects `mainWorldInsertFunction`,
which is defined in `lib/insertion/main-world.ts` and bundled in the package.
It runs only when the user presses Replace, and only when the four preceding
insertion strategies have all failed to write to the editor.

### 2. Keys are unreachable from content scripts — `lib/storage/credentials.ts`

That module is the only reader of API keys and runs exclusively in the
background. `eslint.config.js` carries a `no-restricted-imports` rule scoped to
`entrypoints/content*` that fails the build if it is ever imported there. The
options page never receives a key back either — `publicConnections()` returns
`hasKey: boolean` and never the value, which is asserted by a test in
`tests/storage.test.ts` and again end-to-end in `e2e/options.spec.ts`.

Keys are written to `storage.local`, never `storage.sync`.

### 3. Network destinations — `wxt.config.ts`, `lib/providers/registry.ts`

`host_permissions` is a fixed list of nine provider API hosts.
`optional_host_permissions` exists solely for the "Custom (OpenAI-compatible)"
provider, is not granted at install, and is requested for one specific host
inside the click handler that saves it (`entrypoints/options/main.ts`).

Every outbound URL is built by `endpointFor()` in
`lib/providers/openai-compat.ts` from the registry entry plus, for that custom
provider only, the base URL the user typed.

### 4. Nothing happens without user action — `entrypoints/background.ts`

Provider calls originate from exactly three places: the injected button, the
`enhance-prompt` command (Alt+E), and the context-menu item. There is no timer,
no alarm, no `onInstalled` network call, and no background polling. A soft daily
cap (`lib/providers/softcap.ts`) additionally bounds total requests.

### 5. No HTML injection — `lib/ui/host.ts`

The injected UI is built entirely with `createElement` / `textContent` /
`append` through the `el()` helper. There is no `innerHTML`,
`outerHTML`, `insertAdjacentHTML`, or `document.write` anywhere in the codebase,
which is what allows the content script to run on sites enforcing
`require-trusted-types-for 'script'`. `e2e/hardening.spec.ts` loads the content
script on a page with that CSP and asserts zero violations.

## Third-party dependencies at runtime

Three, and only these three, ship inside the extension bundle:

| Package | Why |
| --- | --- |
| `@floating-ui/dom` | Positions the preview panel so it flips and shifts to stay on screen |
| `zod` | Validates everything read from storage and every user-supplied import |
| `webextension-polyfill` (via WXT) | Promise-based browser API, for Firefox/Chrome parity |

Everything else in `package.json` is a build or test dependency and is not part
of the shipped package.
